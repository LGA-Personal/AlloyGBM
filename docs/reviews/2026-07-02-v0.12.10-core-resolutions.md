## 1. Accuracy

### 1.1 Split-search enforcement for leaf row/hessian constraints

**Fixed in pending v0.12.11 PR.** Split scanners now enforce both child row-count and
child-hessian feasibility before selecting the best candidate. `SplitSelectionOptions` carries
`min_rows_per_leaf`, the trainer threads the active `IterationControls` value into each tree-build
dispatch, and the CPU numeric, categorical, MorphBoost, and PL scanners reject candidates whose
NaN-routed child sides fall below the row threshold. The tree builders still keep their
post-partition row-count checks as defensive validation if a backend violates the split contract.

- `crates/backend_cpu/src/tests/main.rs::categorical_split_scanner_skips_candidates_below_min_rows_per_leaf`
- `crates/engine/src/tests/main.rs::split_scanner_min_rows_falls_back_to_feasible_split`

### 1.2 Default quantile binning

**Fixed in pending v0.12.11 PR.** The Python estimator default for
`continuous_binning_strategy` is now `"quantile"` for `GBMRegressor` and inherited
`GBMClassifier` / `GBMRanker` fits. `MultiLabelGBMRanker` also defaults its continuous-binning
property and joint-mode metadata to `"quantile"` so both independent and joint modes follow the
same default. Explicit `"linear"` and `"rank"` settings remain available.

The joint trainer now threads `continuous_binning_strategy` and `continuous_binning_max_bins`
through `train_joint_multi_label_ranker` instead of hard-coding equal-width (`Linear`) binning.
Previously joint training always binned continuous features linearly while the prediction path
quantized them with the configured strategy; under the new quantile default this produced a
train/predict binning mismatch that corrupted joint predictions on skewed features. The redundant
`max_bin` bridge argument was removed, and `continuous_binning_max_bins` falls back to the legacy
`max_bin` kwarg so joint fits that only set `max_bin` keep a consistent train/predict bin count.

The default switch also exposed and fixed a multiclass warm-start offset bug: resumed multiclass
fits now infer completed rounds from the maximum encoded tree id across all class stump streams,
not only class 0. This prevents continuation from reusing tree ids when one class stops producing
splits before another under quantile binning.

### 1.5 PL path-feature regressors

### 1.6 Training-loss acceptance gate

**Fixed in pending v0.12.11 PRs.** DART is exempt from the hard training-loss
acceptance gate because dropout normalization is intentionally non-monotone. Ranking objectives
were already exempt. A follow-up PR made training-loss stopping opt-in globally by adding
`IterationControls::with_training_loss_gate()`, leaving the default and auto policy at
`min_loss_improvement = 0`, `max_consecutive_weak_improvements = 0`, and
`training_loss_gate_enabled = false`. MorphBoost warmup still skips empty or slightly
negative-improvement warmup rounds so the LR ramp can progress.

### 2.2 Release the GIL

**Partially fixed in [PR #78](https://github.com/LGA-Personal/AlloyGBM/pull/78) and follow-up
PRs.** Native prediction paths now release the Python GIL around Rust compute via
`Python::detach`, including `NativePredictorHandle`, the standalone predictor bridge functions,
and the joint multi-output predictor handle. Native SHAP bridge calls now also detach around
explanation and global-importance compute, including scalar, multi-output, dense, and
predictor-aligned binning entry points.

Native training bridge calls now detach around the Rust fit, including fits that use Python
custom objectives, custom losses, or custom metrics. Python argument parsing and bytes/list
materialization still happen before detaching; `CustomPythonObjective` and
`CustomPythonMetricCallback` reattach with `Python::attach` only while invoking the Python
callable and extracting its result.

### 2.9 Move quantile-cut derivation out of Python

## 3. Inference performance

**Partially fixed in pending v0.12.11 PR.** Predictor tree loading now validates split feature
indices once at artifact load, collapses parent-relative stump deltas into tree-weighted absolute
branch contributions, and resets each loaded tree's runtime weight to `1.0` after the DART weight
has been folded in. Single-output and multiclass prediction loops now descend each tree to the
terminal selected branch and add one contribution per tree instead of accumulating every visited
node. Objective post-transform dispatch is cached as a small enum at load time instead of
string-matching `metadata.objective` for every prediction row.

Because this intentionally changes f32 addition ordering inside a tree, predictor/engine parity
tests now use a tight `1e-6` tolerance for trained multi-level fixtures rather than exact
bitwise equality.

Regression coverage:

- `crates/predictor/src/lib.rs::tests::predictor_collapses_same_tree_scalar_deltas`
- `crates/predictor/src/lib.rs::tests::predictor_collapses_dart_weighted_same_tree_deltas`
- `crates/predictor/src/lib.rs::tests::predictor_rejects_split_feature_index_at_load`

Still open: compact node arrays, side tables for rare categorical/linear data, and remaining
prediction API allocation work outside the numeric estimator `predict(...)` surface.

## 4. Python/API work

**Partially fixed in pending v0.12.11 PR.** `GBMRegressor.predict(...)` and
`GBMRanker.predict(...)` now return one-dimensional NumPy arrays across native-handle, dense,
quantized, and canonical fallback paths instead of wrapping every result in `list(...)`. This
keeps the native `predict_numpy` fast path as an array and removes per-row Python float boxing
from the public numeric prediction API. `GBMRegressor.predict_from_artifact(...)` remains a
list-returning compatibility helper, and classifier label prediction remains out of scope for
this PR.

**Further narrowed in pending v0.12.11 PR.** Public Python metric helpers now coerce through
NumPy arrays instead of list materialization and compute with vector operations for regression
metrics (`rmse`, `mae`, `r2_score`), classification metrics (`accuracy`, `log_loss`), finance
metrics (`pearson_correlation`, `rank_ic`, `hit_rate`, `icir`), ranking `ndcg`, and GLM deviance
metrics (`poisson_deviance`, `gamma_deviance`, `tweedie_deviance`). Existing list inputs,
NumPy arrays, and pandas-like adapters remain accepted, and the validation behavior for empty,

## 5. Robustness / validation

**Fixed in pending v0.12.11 PRs.** `ModelMetadata` JSON decoding is now key-driven for
the current metadata schema, so artifacts no longer depend on the serializer's exact field order.
Required fields (`format_version`, `feature_names`, `trained_device`) are validated by key, while
`objective` and `num_classes` keep their existing backward-compatible defaults.

Unknown metadata fields are skipped as JSON values instead of rejected, so future metadata additions
do not break older readers when the current required schema fields are still present and valid. The
final follow-up replaces the remaining hand-rolled parser/serializer with a private `serde_json`
DTO, preserving unknown-field tolerance, duplicate known-field rejection, `objective` back-compat,
and omission of `num_classes` when it is absent.

**Partially fixed in pending v0.12.11 PRs.** Predictor artifact loading now bounds the per-tree
heap-style node slot array before allocation. Artifacts whose sparse local node IDs would require
more than `MAX_TREE_NODE_SLOTS` (65,536) predictor node slots for a single tree are rejected with a
contract violation instead of constructing a large mostly-empty `nodes_by_local_id` array.

The limit is a single shared constant, `alloygbm_core::MAX_TREE_NODE_SLOTS`, enforced on **both**
sides of the train/predict boundary so the contract is symmetric: the trainer's
`encode_tree_node_id` rejects any split whose heap child would land at slot
`>= MAX_TREE_NODE_SLOTS`, so a fit that would exceed the budget fails fast with a clear error
rather than emitting an artifact the predictor cannot load. This closes the asymmetry flagged in
review — previously the trainer permitted local node IDs up to the encoding stride (`1 << 20`),
16× larger than the predictor's cap, so a deep-but-legal tree could train yet fail to load. The
stride (`1 << 20`) is retained purely as the per-tree id multiplier; the slot budget is the
tighter, deliberately conservative bound (≈ depth 16, far more leaves than any realistic tree).

Artifact loading now also rejects excessive declared section counts and per-section payload
lengths before descriptor parsing or payload copying. The serializer and `ModelIoContractV1`
validator enforce the same limits so locally emitted artifacts cannot exceed the loader budget.

Additional artifact hardening remains open, including bin-count limits and broader metadata
validation.
