# AlloyGBM — Core Review: Recommended Improvements

| Date | Reviewer | Version reviewed | Commit | Status |
|---|---|---|---|---|
| 2026-07-02 | Claude Fable 5 | v0.12.10 | `cdc5bf0` | [In progress](2026-07-02-v0.12.10-core-resolutions.md) |

Scope: design, efficiency, and robustness of the full workspace (6 Rust crates + PyO3
bindings + Python estimator layer), with emphasis on making training and inference faster
**and** more accurate. Baseline health at review time: `cargo test --workspace --release`
passes; the Python suite passes (see §7).

Findings are ranked within each section. Items marked **[measured]** are backed by the A/B
experiments in §0.

> **Companion doc:** the [special-modes review](2026-07-02-v0.12.10-special-modes.md) covers the
> opt-in modes (MorphBoost, DRO leaves, PL trees, factor neutralization, GLM/quantile/ranking
> objectives, DART/GOSS) — including **two measured correctness bugs**: subsampled/GOSS fits
> never apply committed trees to unsampled rows (model divergence), and DART is aborted at
> round 2 by the training-loss gate.

## 0. A/B evidence gathered during this review

California housing (15,480 train / 5,160 test rows, 8 features), 200 estimators,
`learning_rate=0.1`, `max_depth=6`, this machine (arm64 macOS):

| Configuration | Fit time | Test RMSE | Rounds completed |
|---|---|---|---|
| AlloyGBM **defaults** (`linear` binning, `auto` policy) | 0.05 s | **0.5594** | **37/200** — stopped: `LeafRowsBelowThreshold` |
| AlloyGBM `quantile` binning, `auto` policy | 0.17 s | 0.4821 | 200/200 |
| AlloyGBM `quantile` binning, `manual` policy | 0.24 s | **0.4568** | 200/200 |
| LightGBM 4.6 (`num_leaves=63`) | 0.81 s | 0.4578 | 200/200 |
| XGBoost 3.2 (`hist`) | 0.29 s | 0.4603 | 200/200 |

Two conclusions drive the top recommendations:

1. **The engine is already competitive** — with quantile binning and manual policy it beats
   LightGBM on accuracy at ~3× faster fit on this dataset.
2. **The out-of-the-box configuration throws that away** — defaults are 22% worse RMSE than
   what the same library achieves with two kwarg changes, and training silently aborts at
   round 37.

## 1. Accuracy — modeling improvements (highest impact)

### 1.1 Enforce `min_rows_per_leaf` / `min_child_hessian` *inside* the split search, not post-hoc **[measured]**

`build_tree_level_wise` / `build_tree_leaf_wise` ([tree_build.rs:293](../../crates/engine/src/trainer/tree_build.rs:293))
select the single gain-argmax split, **then** partition rows, **then** reject the node if a
child violates `min_rows_per_leaf`. Three problems compound:

- A valid second-best split is never considered — the node is dropped entirely (`continue`).
- When this happens at the **root**, `candidate_round_stumps` comes back empty and the round
  loop **breaks, terminating the whole fit** ([trainer/mod.rs:2251](../../crates/engine/src/trainer/mod.rs:2251)).
  This is exactly what the default-config benchmark shows: `stop_reason_ = LeafRowsBelowThreshold`
  at round 37/200.
- The rejected split's partitioning work (a full parallel row scan) is wasted.

The split scan already has per-side `count` and `hess` for every candidate
(`best_split_for_feature_standard_simd` computes `eff_lc`/`eff_rc` per lane — [lib.rs:627](../../crates/backend_cpu/src/lib.rs:627)).
Adding `count >= min_rows_per_leaf` to the SIMD validity mask (and the scalar/categorical
scaffolds) makes every returned candidate feasible, recovers the second-best split, and turns
the post-partition check into a debug assertion. This is how LightGBM enforces
`min_data_in_leaf`. **Expected effect: strictly better accuracy under any non-trivial
`min_rows_per_leaf`, and no more silent training aborts.** Same argument applies to the
monotone and leaf-magnitude rejections (see 1.4).

### 1.2 Make `quantile` the default `continuous_binning_strategy` **[measured]**

The default is equal-width `linear` binning ([_core.py:47](../../bindings/python/alloygbm/_regressor/_core.py:47)).
Equal-width bins collapse under skewed/heavy-tailed features (most rows land in a few bins,
which also aggravates 1.1). Every mainstream GBDT library defaults to density/quantile-based
binning. Measured: 0.5594 → 0.4821 RMSE from this single change. Keep `linear` available for
the latency-sensitive predict-time quantization path, but default fit-time binning should be
quantile (or a hybrid: quantile cuts with equal-width fallback for near-uniform features).

### 1.3 Recalibrate (or soften) the `auto` training policy **[measured]**

`auto` costs 0.482 vs 0.457 RMSE against `manual` on the same data, and its
`min_rows_per_leaf` heuristic is what triggered the round-37 abort under linear binning.
Recommendations:
- Re-tune the dataset-aware heuristics on the benchmark suite (`benchmarks/`) rather than
  fixed thresholds; log the derived values in `training_summary_` so users can see them.
- After 1.1 lands, the same heuristics become much less dangerous (they constrain the search
  instead of aborting it) — re-benchmark then.
- Consider `training_policy="manual"` as default until auto demonstrably wins on the suite.

### 1.4 True monotone-constraint enforcement (bounds propagation)

Current enforcement compares left/right leaf values post-hoc and **rejects the node**
([tree_build.rs:358](../../crates/engine/src/trainer/tree_build.rs:358)). This (a) discards the node
instead of trying other splits/features, and (b) does not actually guarantee monotonicity of
the ensemble — a descendant split can still produce a locally-ordered pair that violates the
constraint relative to a cousin subtree. The standard fix (LightGBM-style) is to propagate
`[min, max]` bounds down each path and clamp leaf values to the mid-point at constrained
splits. This is both more correct and higher-accuracy (no wholesale node rejection).

### 1.5 PL (linear) leaves regress on `features[0..8]` — use path features instead

Both tree builders pass `regressor_features = (0..min(F, 8))`
([tree_build.rs:391](../../crates/engine/src/trainer/tree_build.rs:391)). Column *order* determines
which features get linear terms — reordering X changes the model. LightGBM's `linear_tree`
uses the features on the root-to-leaf split path (which are, by construction, the locally
predictive ones). Using path features (capped at `MAX_PL_REGRESSORS`) should improve
`leaf_model="linear"` accuracy and remove the column-order sensitivity. At minimum, document
the current behavior loudly.

### 1.6 Make the training-loss acceptance gate opt-in

The round loop rejects a round (and can stop the fit) when *training* loss fails to improve
([trainer/mod.rs:2353](../../crates/engine/src/trainer/mod.rs:2353)), with a weak-improvement-streak
counter. Under `row_subsample`/GOSS noise, small learning rates, or DART, non-monotone training
loss is normal and not a reason to stop; no mainstream GBDT gates on training loss (ranking is
already exempted for exactly this reason). Recommend: keep the gate for MorphBoost warmup where
it has a purpose, but default `min_loss_improvement`-style stopping to off, relying on
validation early stopping. This also deletes one full O(N) loss pass per round when off.

### 1.7 Low-hanging accuracy features

- **Exclusive Feature Bundling (EFB)** for one-hot/sparse inputs — large real-world speed +
  modest accuracy win; fits the existing binning layer.
- **Per-feature quantile sketch on subsample** for very large N (current quantile-cut
  derivation sorts full columns; a sketch makes quantile binning cheap enough to be default
  everywhere).
- **`colsample_bynode`/`bylevel`** — you have per-round `col_subsample`; per-node sampling is
  a cheap addition with known accuracy benefits on wide data.
- **Goss `top_rate`/`other_rate` exposure** if not already user-tunable from Python.

## 2. Training performance (Rust core)

### 2.1 Add a tuned `[profile.release]` to the workspace `Cargo.toml`

There is none today, so wheels ship with defaults (16 codegen units, no LTO):

```toml
[profile.release]
lto = "thin"          # or "fat" for dist builds; measure
codegen-units = 1
```

For numeric kernels this is typically a free 5–15%. Keep `panic = "unwind"` (PyO3 needs it to
convert panics to Python exceptions). Consider a `[profile.dist]` used by the release CI, and
building x86-64 wheels with a `x86-64-v3` target-cpu variant (the `wide` crate only emits AVX2
when the target supports it — a portable baseline build compiles the f32x8 paths to SSE2).

### 2.2 Release the GIL around training, prediction, and SHAP

There is **no `allow_threads`/`detach` anywhere in `bindings/python/src/`**. Every fit and
predict holds the GIL for its entire duration — a multi-second fit freezes all other Python
threads (servers, notebooks with progress UIs, thread-pool inference). Wrap the compute-heavy
Rust calls in `py.detach(|| …)` (pyo3 0.29). Two carve-outs: paths that invoke
`CustomPythonObjective` / `CustomPythonMetricCallback` must re-attach around the callback, and
borrowed numpy views must be materialized (or kept as `PyReadonlyArray` guards, which is
allowed) before detaching.

### 2.3 Slim the histogram bin: gate `grad_sq_sum` on demand, go SoA end-to-end

`HistogramBin` is 16 B (`grad, hess, grad_sq, count`) and the innermost kernel does 4
read-modify-writes per (row, feature) ([lib.rs:370](../../crates/backend_cpu/src/lib.rs:370)).
`grad_sq_sum` is only consumed when DRO is active (`leaf_effective_gradient` ignores it
otherwise). Recommendations, in increasing ambition:

1. Compile two accumulation kernels (with/without grad_sq) and select per fit — removes a
   multiply + add + 4B of cache traffic per cell on the standard path (~25% of accumulator work).
2. Keep the SoA arena (`grad_sums/hess_sums/counts` arrays) all the way through split
   finding instead of materializing AoS `Vec<FeatureHistogram>` per node
   (`materialize_tile_histograms` allocates a fresh `Vec<HistogramBin>` per feature per node).
   The prefix scan in the split kernel is trivially SoA-friendly, subtraction becomes three
   `slice` loops, and per-node allocation churn disappears.
3. Interleave `(grad, hess)` as f32 pairs (LightGBM layout) so one accumulation touches one
   cache line.

### 2.4 Histogram/partition allocation churn → per-tree pools

- `subtract_histogram_bundle` allocates a fresh bundle per split (the `_into` variant exists
  but the tree builders call the allocating one — [tree_build.rs:519](../../crates/engine/src/trainer/tree_build.rs:519)).
- `apply_split_with_stats_parallel` allocates per-chunk left/right Vecs and then concatenates
  ([lib.rs:1427](../../crates/backend_cpu/src/lib.rs:1427)). The standard alternative is a single
  per-tree `row_indices` buffer partitioned stably in place (LightGBM's `DataPartition`):
  zero allocation per split and better locality for the subsequent child histogram build.
- The SIMD split scan allocates 4 cumulative vectors per feature per node
  ([lib.rs:514](../../crates/backend_cpu/src/lib.rs:514)) — reuse a thread-local scratch like
  `THREAD_ARENA`.

### 2.5 Parallelize across nodes within a level

`build_tree_level_wise` iterates nodes at a level sequentially; parallelism only exists inside
each node's histogram build (tile-level) and split scan (feature-level). Deep in the tree,
nodes are small and per-node parallelism stops paying (`PARALLEL_TILE_WORKLOAD_THRESHOLD`),
while node-level parallelism is abundant. Building histograms for all nodes of a level in one
`par_iter` (then resolving splits) is the classic fix, and composes with the sibling
subtraction trick (build all "smaller" children in parallel, subtract for the others).

### 2.6 Drop the duplicate row-major bin storage

`BinnedMatrix::new` always builds **both** row-major and column-major copies
([binned.rs:77](../../crates/core/src/binned.rs:77)). Histogram building and split application both
prefer the col-major path when present; the row-first arena kernel only runs when col-major is
absent. Storing only col-major halves binned-matrix memory (the dominant training allocation
alongside the raw f32 values) and removes a full transpose+copy at fit start. If the row-first
kernel wins on some shapes, make the duplicate opt-in by heuristic rather than unconditional.

### 2.7 Cut fixed per-round O(N) overheads

Each round does: full `candidate_predictions.copy_from_slice(&predictions)`, a full
training-loss pass, and (per 1.6) sometimes a second copy for DART backups. With 1.6 opt-in,
the loss pass disappears; the candidate-copy can be replaced by tracking the per-round delta
(the tree walk already knows exactly which rows changed and by how much — apply/rollback).

### 2.8 Expose thread control (`n_jobs`)

Rayon's global pool is used implicitly; there is no `n_jobs` on the estimators and no
documented env var. Add `n_jobs` (build a scoped `ThreadPool` per fit, or set the global pool
once) — essential for co-tenancy with other parallel libraries and for reproducible
benchmarking.

### 2.9 Move quantile-cut / sorted-values derivation out of Python

`_derive_dense_sorted_feature_values` / `_derive_dense_feature_quantile_cuts`
([_quantization.py:248](../../bindings/python/alloygbm/_regressor/_quantization.py:248)) extract each
column with a Python list comprehension and sort in the interpreter — O(N·F) Python-object work
before the Rust fit even starts, on the strategy you want to make the default (1.2). Either
numpy-vectorize (`np.sort` on a transposed array) or push into the Rust `quantization.rs`
(rayon per feature). This matters more once quantile is the default.

## 3. Inference performance

### 3.1 Replace `Vec<Option<PredictorTreeNode>>` with a compact node array

Each tree stores heap-indexed (2i+1/2i+2) `Option<PredictorTreeNode>` where the node struct
carries `usize` feature index, two `Option<LinearLeafCompact>` (Vecs), and an
`Option<Vec<u8>>` bitset — roughly 150 B per slot plus pointer-chasing, and the vector length
is 2^(depth+1) regardless of how sparse the tree is. Two costs:

- **Cache behavior**: batch prediction walks ~150 B nodes scattered across 2^depth slots.
  A 16–24 B SoA node (u32 feature | f32 threshold | u8 flags | child/leaf index) with side
  tables for the rare categorical bitsets and linear leaves is the standard layout and
  typically 2–5× faster on batch predict.
- **Memory blow-up on deep leaf-wise trees**: `max_leaves=255` with unconstrained depth can
  legally produce depth ≫ 20; 2^20 slots × ~150 B ≈ 150 MB *per tree* for a 255-leaf tree.
  This is also a robustness concern when loading untrusted artifacts (see 5.2).

### 3.2 Pre-collapse stump deltas at load time — one add per tree instead of one per level

The model form stores parent-relative *deltas* at every visited node, so `predict_row`
accumulates `tree_weight * leaf` at **every level** of the path
([lib.rs:543](../../crates/predictor/src/lib.rs:543)). Since the walk's total contribution equals the
*absolute* leaf value at the last visited node (deltas telescope), the loader can precompute
absolute values per node (and fold `tree_weight` in) so prediction descends to the deepest
existing node and does **one** add. That removes D−1 FMAs and the per-level `tree_weight`
multiply per tree — a direct constant-factor cut on every prediction, with no output change.

### 3.3 Hoist per-node checks out of the walk loop

- The `node.feature_index >= feature_count` check (with `format!` error construction on the
  cold side) runs per node per row ([lib.rs:530](../../crates/predictor/src/lib.rs:530)). Validate
  all feature indices once at artifact load; the walk then indexes unconditionally (safe —
  bounds still checked by the slice index).
- `post_transform` string-matches `metadata.objective.as_str()` **per row**
  ([lib.rs:766](../../crates/predictor/src/lib.rs:766)). Resolve to an enum once at load.

### 3.4 Return numpy from `predict`, end-to-end

`GBMRegressor.predict` is annotated `-> list[float]` and does `list(predict_numpy(arr))`
([_core.py:2434](../../bindings/python/alloygbm/_regressor/_core.py:2434)) even though
`predict_numpy_array` (Vec→`PyArray1`, no list materialization) already exists on the handle.
For 1M rows this is 1M PyFloat allocations plus a second pass for the user's inevitable
`np.asarray`. Returning `ndarray` also matches the sklearn contract (estimators return arrays,
not lists) — worth the minor API-change note in the changelog.

### 3.5 Kill the remaining copy in the bytes predict path

`predict_batch_dense_bytes` decodes f32s into a per-chunk row buffer
([lib.rs:728](../../crates/predictor/src/lib.rs:728)). Since the Python layer can always produce a
C-contiguous float32 ndarray, prefer routing everything through `predict_numpy_array`
(zero-copy borrow) and reserve the bytes path for the no-numpy fallback only. Combined with
2.2 (GIL release) this makes batch inference allocation-free up to the output vector.

## 4. Python layer

### 4.1 Vectorize `evaluation.py`

Every metric (`rmse`, `mae`, `r2_score`, `ndcg`, `log_loss`, IC helpers) is a pure-Python
loop/genexp over zipped floats. These run per-round when used as eval metrics and over full
predictions in user code. Rewriting on numpy is ~50–200× faster and *less* code. Keep the
list-input tolerance via a single `np.asarray` at the top.

### 4.2 Retire the `tolist()` "fast path" for data transfer

`_flatten_native_matrix_candidate` ([_quantization.py:184](../../bindings/python/alloygbm/_regressor/_quantization.py:184))
turns numpy arrays into Python `list[float]` (N×F boxed floats) before PyO3 re-converts
element-wise. The bytes payload path already avoids this for the main dense fit; audit the
remaining callers (categorical path, validation matrices, joint bridge, SHAP entry points) and
move them to bytes or, better, `PyReadonlyArray2<f32>` parameters (true zero-copy, no
`tobytes()` duplicate of the whole matrix).

### 4.3 sklearn conformance sweep

Beyond `predict` returning a list (3.4): run `sklearn.utils.estimator_checks` in CI
(`parametrize_with_checks`) — it will also catch `get_params`/`set_params`/clone edge cases in
the growing kwarg surface, cheaply guarding the five-place update rule in CLAUDE.md.

## 5. Robustness & code quality

### 5.1 Replace the hand-rolled positional JSON parser for `ModelMetadata`

`artifact_format.rs` (1,710 lines) hand-parses metadata positionally; CLAUDE.md itself calls it
"very brittle — adding fields requires careful ordering". `serde_json` is a small, ubiquitous,
`forbid(unsafe)`-compatible dependency; with `#[serde(default)]` fields become order-independent
and forward-compatible. Keep the binary section format as-is (it's versioned and fine) — only
the JSON header needs rescuing. If a zero-dep policy is the blocker, at minimum make the parser
key-driven rather than positional.

### 5.2 Validate artifacts at load against resource blow-ups

Related to 3.1: a crafted (or just deep leaf-wise) artifact can request 2^depth node slots per
tree. Bound accepted local node ids / depth at decode time and return a contract error instead
of attempting a multi-GB allocation. Also applies to declared bin counts and section lengths
(cross-check against payload sizes before `Vec::with_capacity`-style allocations).

### 5.3 Repo hygiene: stale worktrees

`.worktrees/` and `.claude/worktrees/` contain full stale copies of the codebase (old
`quantile-regression`, `refactor-*`, `bugfix-v0.12.4` trees). They inflate clones/searches and
show up in greps (they dominated the `wc -l` survey in this review). Prune merged worktrees and
add both directories to `.gitignore`/exclude if not already.

### 5.4 Benchmark regression tracking in CI

`backend_cpu` has a criterion bench (`histogram_kernels`) and `benchmarks/` has a rich
cross-library harness, but nothing runs in CI. Add a small, fixed-seed benchmark job (train +
predict wall time and RMSE on 2–3 datasets) that posts deltas on PRs — this repo ships
optimization passes regularly (v0.12.10) and has byte-equivalence tests, but no guard against
*performance* regressions.

### 5.5 Bounds-check pressure under `forbid(unsafe)` — structure around it

The policy is a genuine strength; keep it. But the hottest loop indexes
`arena.grad_sums[idx0..idx7]` etc. with computed indices, each bounds-checked. Where profiles
show it matters, restructure so the compiler can elide checks safely (e.g., take
`&mut [f32]` sub-slices of exactly `bin_count` per feature before the row loop, or use
`chunks_exact_mut`), rather than reaching for `unsafe`. Verify with `cargo asm`/criterion
rather than assuming.

## 6. Suggested priority order

| # | Item | Type | Effort | Expected payoff |
|---|------|------|--------|-----------------|
| 1 | 1.1 in-search `min_rows_per_leaf` | accuracy + robustness | M | Removes silent training aborts; strictly better splits |
| 2 | 1.2 default quantile binning | accuracy | S | −14% RMSE on the measured dataset, out of the box |
| 3 | 2.1 release-profile tuning | speed | S | ~5–15% across the board, free |
| 4 | 2.2 GIL release | usability/throughput | S–M | Unblocks threaded serving; no more frozen notebooks |
| 5 | 3.4 numpy `predict` return | speed + API | S | Removes O(N) boxing on every predict |
| 6 | 1.3 auto-policy recalibration | accuracy | M | Closes the 0.482→0.457 gap for default users |
| 7 | 3.1 + 3.2 predictor node layout + delta collapse | inference speed | M–L | Multi-× batch predict; bounds deep-tree memory |
| 8 | 2.3 + 2.4 SoA histograms + partition pools | training speed | L | Biggest core-training win; sets up 2.5 |
| 9 | 1.4 monotone bounds propagation | accuracy/correctness | M | True monotonicity guarantee |
| 10 | 5.1 serde metadata | robustness | M | De-risks every future field addition |

Items 1–5 are individually small, independent, and each is verifiable with the existing test
suite plus the benchmark script used for §0.
