## 1. Critical correctness issues

### 1.1 Row subsampling / GOSS committed predictions

**Fixed in pending v0.12.11 PR by `9d93ad6`.** The single-output and multiclass training loops now rebuild
candidate training predictions by walking the accepted tree over every training row after tree
construction. This keeps the committed training state consistent with artifact inference for
uniform row subsampling, GOSS, DART, missing-value routing, native categorical routing, and PL
leaf evaluation.

Regression coverage:

- `bindings/python/tests/test_goss.py::test_uniform_row_subsample_updates_all_training_rows_after_commit`
- `bindings/python/tests/test_goss.py::test_goss_regressor_predictions_do_not_overshoot_simple_target_range`

Synthetic A/B on this branch:

| Mode | Before | After |
|---|---:|---:|
| `row_subsample=0.5`, train RMSE | 1.61347 | 0.09126 |
| `boosting_mode="goss"`, train RMSE | 1.59549 | 0.07661 |

Both after-fix runs completed all 120 requested rounds with `stop_reason_ =
"CompletedRequestedRounds"`.

### 1.2 DART training-loss gate

**Fixed in pending v0.12.11 PR by `9d93ad6`.** The training-loss acceptance gate now exempts DART in the
same spirit as ranking objectives. DART dropout and normalization are non-monotone by design,
so a higher training loss in a round is not a valid reason to discard the round or terminate
the fit.

Regression coverage:

- `bindings/python/tests/test_dart.py::test_dart_regressor_does_not_stop_at_first_dropout_round`

Synthetic A/B on this branch:

|---|---:|---:|
| `boosting_mode="dart"`, completed rounds | 8 / 80 | 80 / 80 |

## 5. Factor neutralization

**Partially fixed in pending v0.12.11 PR.** Objective-dependence is now documented: per-round
gradient neutralization projects gradients while leaving Hessians unchanged, so non-constant
Hessian objectives do not produce a simple projection of unneutralized leaf values. Fitted scalar
estimators and joint `MultiLabelGBMRanker` now add post-fit prediction exposure diagnostics to
`factor_exposure_diagnostics_`: `prediction_exposure_dot`, `prediction_exposure_abs`, and
`prediction_exposure_l2`, computed from transformed fit-time exposures and fitted training
predictions. Multiclass models have no single scalar prediction vector, so they keep the
transform/means/stds metadata but omit the prediction-exposure entries.

Regression coverage:

- `bindings/python/tests/test_factor_neutralization.py::FactorNeutralizationTests::test_factor_diagnostics_report_post_fit_prediction_exposure`
- `bindings/python/tests/test_factor_neutralization.py::FactorNeutralizationTests::test_multiclass_neutralization_fit_skips_prediction_exposure`
- `bindings/python/tests/test_joint_multilabel.py::test_joint_factor_diagnostics_report_post_fit_prediction_exposure`

Still open: projection parallelism, categorical prefix sums, default exposure standardization for
split penalties, and scratch reuse.

### 6.2 GLM stabilizers and f64 accumulators

**Partially fixed in pending v0.12.11 PR by `97ed521` and follow-up PR by
[`abc5ef7`](https://github.com/LGA-Personal/AlloyGBM/commit/abc5ef7) (merged with clippy-cleanup
[`4b58f7c`](https://github.com/LGA-Personal/AlloyGBM/commit/4b58f7c)).** GLM initial predictions
now compute weighted target means with f64 accumulators, and Poisson/Gamma/Tweedie objective
losses now accumulate telemetry in f64 before returning f32. The Poisson objective also inflates
its Newton hessian by `exp(0.7)` by default, matching the standard Poisson max-delta-step
stabilizer, and exposes `poisson_max_delta_step` on `GBMRegressor`, `GBMRanker`, independent
and joint `MultiLabelGBMRanker`, and the native Rust `TrainParams`/objective path.

The tunable step is carried as a `max_delta_step` field on `PoissonObjective` itself (defaulting
to 0.7 via `Default`); the initial parallel `TunedPoissonObjective` type was removed in `4b58f7c`
because its `PoissonObjective::new(..)` returned a different type and tripped
`clippy::new_ret_no_self` under `-D warnings`. `new(..)` now returns `Self`.

Regression coverage:

- `crates/engine/src/tests/morph_state.rs::glm_initial_prediction_accumulates_mean_in_f64`
- `crates/engine/src/tests/morph_state.rs::poisson_gradient_uses_stabilized_hessian`
- `crates/engine/src/tests/morph_state.rs::poisson_gradient_hessian_stabilizer_is_tunable`
- `bindings/python/tests/test_regressor_contract.py::GBMRegressorContractTests::test_constructor_rejects_invalid_values`
- `bindings/python/tests/test_regressor_contract.py::GBMRegressorContractTests::test_get_params_and_set_params_roundtrip`
- `crates/engine/src/tests/morph_state.rs::glm_losses_accumulate_in_f64`

Still open: broad benchmark validation on skewed count data.

### 6.4 Multiclass softmax

**Partially fixed in [PR #74](https://github.com/LGA-Personal/AlloyGBM/pull/74) by
[`d913e07`](https://github.com/LGA-Personal/AlloyGBM/commit/d913e07).** `GBMClassifier` now rejects
explicit string objectives consistently at construction and through `set_params`, explaining that
classifier objectives are auto-detected (`binary_crossentropy` for binary labels and
`multiclass_softmax` for multiclass labels). The error message points users with regression-style
objectives such as `quantile` or `poisson` to `GBMRegressor(objective=...)` or grouped
`GBMRanker(ranking_objective=...)` instead. Callable custom objectives remain available for the
existing binary custom-objective path.

Regression coverage:

- `bindings/python/tests/test_classifier_and_metrics.py::GBMClassifierTests::test_rejects_explicit_string_objectives_with_alternative_estimators`
- `bindings/python/tests/test_quantile_objective.py::test_quantile_supported_combinations`
- `bindings/python/tests/test_custom_objective.py::TestClassifierCustomObjective::test_classifier_custom_objective_binary`

Parallel per-class multiclass tree builds remain open.

## 7. Joint multi-output trainer

**Partially fixed in [PR #75](https://github.com/LGA-Personal/AlloyGBM/pull/75) by
[`227cfb8`](https://github.com/LGA-Personal/AlloyGBM/commit/227cfb8).** Joint
`MultiLabelGBMRanker` now validates native-categorical columns before training when
`max_cat_threshold > 0`: category IDs must be integer-valued, dense `0..K-1`, and no joint
native-categorical feature may exceed 64 categories. This makes the joint trainer's `u64`
categorical bitset cap explicit instead of silently letting Rust skip high-cardinality native
categorical splits.

Regression coverage:

- `bindings/python/tests/test_joint_multilabel.py::test_joint_native_categorical_rejects_more_than_64_categories`
- `bindings/python/tests/test_joint_multilabel.py::test_joint_native_categorical_rejects_non_dense_ids`
- `bindings/python/tests/test_joint_multilabel.py::test_joint_native_categorical_rejects_non_integer_values`

The joint-path `row_subsample=0.5` quality regression, joint categorical split-penalty prefix
sums, and per-output gradient parallelism remain open.
