## 1. Critical correctness issues (fix before anything else in this doc)

### 1.1 Row subsampling / GOSS: committed predictions are only updated on sampled rows **[bug, measured]**

**Symptom.** `boosting_mode="goss"` produces train RMSE 2.18 / test 2.19 — *worse than a
constant predictor* (y std ≈ 1.15) — with predictions spanning (−3.1, 8.4) against a target
range of (0.15, 5.0). Plain `row_subsample` degrades monotonically with the sampled fraction
(1.0 → 0.4568, 0.8 → 0.5266, 0.5 → 1.1120) with the same prediction-range blow-out. All 200
rounds "pass" the training-loss gate while the deployed model diverges.

**Mechanism.** In the single-output round loop, the tree builders update
`candidate_predictions` only via `apply_partition_leaf_updates` over the partition row
indices, which descend from `root_row_indices` — the **sampled subset**
([tree_build.rs:467](../../crates/engine/src/trainer/tree_build.rs:467)). The round is then committed
with `std::mem::swap(&mut predictions, &mut candidate_predictions)`
([trainer/mod.rs:2524](../../crates/engine/src/trainer/mod.rs:2524)). No full tree walk ever applies
the new tree to the *unsampled* rows of the training-state predictions.

Consequences: (a) next-round gradients for unsampled rows are computed against stale
predictions, so successive trees keep re-pushing the same residual in the same direction;
(b) at inference the artifact applies **every** tree to **every** row, so those repeated
pushes sum and the model overshoots — worst under GOSS, where the non-top rows are sampled at
`other_rate` and are stale most rounds; (c) the training-loss gate is evaluated on the same
inconsistent state, so it never fires. The validation path does *not* have this bug (it uses
a full `apply_round_stumps_tree_walk` over all validation rows), which means train/validation
metrics silently disagree about what model is being fit.

**Fix.** After a round is accepted, apply the committed tree to the rows *not* in
`root_row_indices` (the validation path's tree-walk helper already does exactly this walk), or
simply walk all rows like Friedman's stochastic gradient boosting / LightGBM / XGBoost do.
Cost is one tree walk over ≤ N rows per round — trivially parallel and already implemented for
validation. Add a regression test: fixed dataset, `row_subsample=0.5`, assert final in-sample
RMSE beats the baseline-mean predictor and roughly matches a `row_subsample=1.0` fit.

**Check the sibling paths.** The joint trainer's round-end add uses
`walk_tree_into_predictions` (full walk), so it appears unaffected — verify with the same
A/B, along with the multiclass loop (`fit_multiclass_iterations_impl`), which shares the
partition-update pattern.

### 1.2 DART is killed by the training-loss gate at round 2 **[bug, measured]**

`boosting_mode="dart"` completed **1 round** (`stop_reason_ = LossImprovementBelowThreshold`)
and shipped a one-tree model (RMSE 1.077). Mechanism: round 2 drops tree 1, fits the residual,
then re-adds everything at normalized weights `1/(k+1)` — by design, the post-normalization
ensemble usually has slightly *higher* training loss than the pre-dropout ensemble. The strict
`loss_improvement < 0 → break` gate ([trainer/mod.rs:2353](../../crates/engine/src/trainer/mod.rs:2353))
therefore terminates almost every DART fit immediately. Ranking objectives are already exempted
from this gate for exactly this "non-monotone by construction" reason; DART needs the same
exemption (or the gate should become opt-in globally, per main doc §1.6). Any DART benchmark
results published before this fix measured a 1–2 tree model.

### 1.3 PL (linear) leaves: ill-conditioned solve on unstandardized features **[measured]**

`leaf_model="linear"` scored 0.9518 vs 0.4568 scalar baseline. Diagnostics: standardizing X
beforehand improves it to 0.6054; raising `lambda_l2` to 10 on raw X gives 0.4771. The default
configuration is numerically fragile, for three compounding reasons:

1. **Unscaled regressors.** California-housing columns span ~5 orders of magnitude
   (median income ≈ 5 vs population ≈ 3,000). `XᵀHX` entries then span ~10 orders, and a
   single ridge `λ` cannot regularize all coefficients sensibly (it crushes small-scale
   features and barely touches large-scale ones).
2. **f32 accumulation.** `xtg`/`xt_hx` accumulate in f32 (`LinearHistogramBin`); with entries
   at ~1e10 magnitude the Cholesky operates near f32 precision limits. The solve only falls
   back to scalar leaves on non-positive-definiteness — a *badly conditioned but PD* system
   sails through and produces garbage coefficients.
3. **Fixed regressor set = first 8 columns** (main doc §1.5), so which features hit the
   conditioning problem is an accident of column order.

Recommendations, in order: (i) standardize regressor columns internally for the solve (scale
by per-feature std computed once at fit start; unscale coefficients after) — this alone likely
recovers most of the gap and makes `λ` meaningful; (ii) accumulate `xtg`/`xtHx` in f64 for the
selected-child solve (v0.12.10 already narrowed how often full PL histograms are built, so the
f64 cost is contained); (iii) add a condition-number (or diagonal-ratio) guard that falls back
to scalar leaves; (iv) switch regressors to path features. Add an accuracy regression test on
a raw-scale dataset — the current suite evidently only pins PL behavior on well-scaled data.

## 2. MorphBoost

The implementation is clean (warmup byte-equivalence with the standard gain is a great
invariant, and `MorphPrecomputed` hoists the per-iteration constants). Critiques:

### 2.1 The blended terms have incompatible units

`gain = c_g · gradient_score + c_i · info_score` ([morph.rs:59](../../crates/backend_cpu/src/morph.rs:59)):
`gradient_score` is an XGBoost gain built from gradient **sums** — it scales linearly with node
row count and with the loss scale — while `info_score` is built from per-side gradient
**means** normalized by the EMA std, i.e. O(1) regardless of node size or target scale.
Consequently the effective weight of the info term collapses on large nodes/datasets and
dominates on tiny ones; `info_score_weight=0.3` means something different at every node depth
and for every dataset. Same for `balance_adjustment`, a fixed −0.5-max additive penalty
competing against gains that can be O(10³). Fix candidates: normalize `gradient_score` by the
parent gain (dimensionless improvement ratio) before blending, or scale the info/balance terms
by node count. Any change here shifts trained models, so gate it behind the existing
ablation harness (`benchmarks/morph_ablation.py`) and pin the winner with A/Bs.

### 2.2 Hardcoded morph-weight horizon

`morph_weight = tanh(iteration / 20)` — the ramp saturates at ~iteration 60 regardless of
`n_estimators` (20 is baked into `MorphPrecomputed::for_iteration`). For 1,000-round fits the
criterion is effectively fully-on for 94% of training; for 50-round fits it never saturates.
Scale the ramp by `total_iterations` (which is already available) or expose it.

### 2.3 Post-warmup split scan is scalar

The morph path bypasses the SIMD bin scan because of `tanh/ln/exp` (comment at
[lib.rs:443](../../crates/backend_cpu/src/lib.rs:443)). Two cheap options short of full
vectorization: (a) the `tanh` is per-*iteration* (already precomputed) and the `exp` only
fires in `balance_adjustment` when `balance_ratio < 0.1` — the only per-candidate
transcendental is one `ln(1 + |g_mean|)` per side, which `wide` can approximate or which can
be computed on the ≤ 8-lane extracted values after a SIMD-computed gradient part; (b) two-pass:
SIMD-scan the standard gain, keep the top-k candidate bins, rescore only those with the morph
formula (approximation — measure ranking overlap first). Measured cost today: 0.62 s vs 0.24 s
baseline, so there is a real 2×+ to recover for morph users.

### 2.4 MorphBoost + quantile refinement interplay

`refine_quantile_leaf_values` re-derives the morph schedule factors (lr × iter-shrinkage ×
depth penalty) independently of the tree builder ([trainer/mod.rs:2255](../../crates/engine/src/trainer/mod.rs:2255)).
Two code paths must stay in lockstep manually; extract one shared `morph_leaf_scale(iter, depth)`
helper so a future schedule change can't diverge them.

## 3. DRO leaves

The math is sound and well-scoped: the effective-gradient shrinkage
`radius · √n · σ(g)` on the gradient *sum* equals a `radius · σ/√n` shrinkage of the mean —
a standard-error-of-the-mean uncertainty ball, which vanishes for large leaves and bites on
small ones. Critiques:

### 3.1 DRO forces the scalar split path — but it's fully vectorizable

`best_split_for_feature` falls back to the scalar scaffold whenever `dro_config` is set
([lib.rs:433](../../crates/backend_cpu/src/lib.rs:433)). Unlike morph, the DRO formula is
polynomial + one `sqrt` — `wide` has lane-wise `sqrt`. Extending the SIMD kernel with an
optional DRO threshold (variance from the already-tracked `grad_sq` lane) removes the measured
~2.4× slowdown for DRO users. This also pairs with main doc §2.3: `grad_sq_sum` accumulation
should be *enabled by* DRO and *skipped* otherwise.

### 3.2 `radius` units are gradient-scale-dependent

The threshold competes with gradient sums, so a good radius depends on the objective and
target scale (0.05 on squared-error with y ∈ [0,5] means something else entirely on ranking
lambdas). Since σ(g) is already in the formula the *dispersion* part is scale-aware, but the
interaction with `l1_alpha` (added linearly into the same threshold) is not. Recommend: document
radius as "per-σ" explicitly, and validate the DRO default (0.05) with the benchmark harness on
noisy-label datasets — the mode's value proposition (robustness) is currently asserted, not
demonstrated anywhere in `benchmarks/`.

### 3.3 Joint-path DRO is leaf-only (split gain not robust) — decide with data

Documented tradeoff (CLAUDE.md v0.10.5): the K-output histogram carries no `grad_sq`, so joint
split *selection* ignores DRO. Before spending the ~1.5× histogram memory, add a benchmark that
measures whether leaf-only DRO on the joint path actually moves out-of-sample robustness; if it
doesn't, deprecate rather than extend.

## 4. PL trees (beyond the §1.3 correctness item)

- **Memory/bandwidth**: each `LinearHistogramBin` carries `xtg[8] + xt_hx[64]` ≈ 288 B/bin —
  ~18× a scalar bin. The v0.12.10 selected-child optimization was the right call; the next
  step (if PL profiling matters) is building linear histograms only for the top-k features by
  standard gain rather than all features during the PL gain scan.
- **NaN semantics**: NaN regressor values contribute 0 to the linear sum — i.e. implicit
  zero-imputation in *raw feature units*, which is only neutral if features are centered.
  After §1.3(i) internal standardization, contribute-0 becomes mean-imputation for free —
  document this as the intended semantics.
- **`MAX_PL_REGRESSORS = 8`** is a hard cap baked into the stride-8 SIMD layout — fine, but
  the Python docs should state it (silently using 8 of 500 features surprises users).
- **Missing A/B guardrail**: `pl_trees_benchmark.py` exists in `benchmarks/` — wire it to CI
  (main doc §5.4) with a raw-scale dataset so §1.3-class regressions can't recur.

## 5. Factor neutralization

Design is coherent (three modes with clear contracts, inert-config byte-equivalence tests,
`factor_exposure_transform` preprocessing since v0.12.10). Critiques:

### 5.1 `per_round_gradient` projects `grad` only — document the objective-dependence

`project_gradient_pairs_in_place` residualizes `grad` and leaves `hess` untouched
([factor.rs:53](../../crates/engine/src/factor.rs:53)). For squared error (hess ≡ 1) that's exact;
for ranking/GLM/binary the Newton step `−g/h` uses a projected numerator over an unprojected
denominator, so leaf values are not the projection of the unprojected leaf values. Probably the
right pragmatic choice — but it should be stated in the docs, and the neutralization
diagnostics should report *post-fit prediction exposure* (‖Fᵀŷ‖ per factor) rather than only
per-round gradient norms, so users can see what they actually bought. That diagnostic is cheap
(one O(N·K) pass at fit end) and is the number quant users care about.

### 5.2 Per-round projection: parallelize and stop allocating

Each round does an O(N·K) RHS accumulation, an O(K²) solve, and an O(N·K) residual pass —
all serial, plus a fresh `Vec::with_capacity(N)` for residuals every round
([factor.rs:63](../../crates/engine/src/factor.rs:63)). Rayon the two N-passes (chunked accumulation
with per-chunk RHS reduce) and reuse a scratch buffer held by the trainer. The Gram build in
`FactorProjector::new` is O(N·K²) serial — same treatment; it runs once per fit (K per-output
times on the joint path).

### 5.3 `split_penalty`: give categorical the same prefix-sum treatment as numeric

v0.12.10 gave numeric candidates per-feature factor prefix sums
(`prepare_numeric_prefix`, one O(rows·K) scan per feature per node). Categorical candidates
still call `factor_split_penalty_for_candidate`, which **rescans all node rows per prefix k per
NaN direction** ([factor_split.rs:148](../../crates/backend_cpu/src/factor_split.rs:148)) —
O(rows · categories · K) per feature per node. Precompute per-*category* factor sums once
(O(rows·K)), then prefix over the Fisher-sorted category order exactly like the numeric path.
Mechanical change, existing parity test structure (`numeric_prefix_penalty_matches_row_scan_*`)
shows how to pin it.

### 5.4 Default `factor_exposure_transform` to `"standardize"` when `split_penalty` is active

The penalty magnitude scales with exposure units; `factor_penalty` is untunable across datasets
otherwise. The transform machinery already exists (v0.12.10) — flip the default for the one
mode whose semantics depend on scale, keep `"none"` elsewhere for backward compatibility.

### 5.5 Scratch allocation churn on the scalar path

`FactorSplitScratch::new` allocates 4–5 Vecs per *feature* per node when a factor context is
active ([lib.rs:805](../../crates/backend_cpu/src/lib.rs:805)). Hoist to a thread-local like
`THREAD_ARENA`. (Factor contexts also force the scalar split path — the prefix-sum design is
actually SIMD-compatible per 8-bin chunk if this mode ever becomes hot.)

## 6. Objectives

### 6.1 LambdaMART / pairwise ranking: the gradient pass needs the standard optimizations

`compute_gradients` ([ranking.rs:394](../../crates/engine/src/objectives/ranking.rs:394)) is a
serial O(Σ n_g²) double loop with **per-pair** `2^label` (`powf`) and `log2` calls:

1. **Precompute per-doc arrays** (`gain[i] = 2^label − 1`, `discount[r] = 1/log2(r+2)`) before
   the pair loop — removes two `powf` + two `log2` per pair, an easy several-× on the gradient
   pass with zero output change.
2. **Parallelize across groups** — groups write disjoint slices; `par_iter` over group
   boundaries with `split_at_mut`-style partitioning is safe and mechanical.
3. **Add a truncation parameter** (LightGBM's `lambdarank_truncation_level`): only pairs
   involving a top-k-by-current-score doc. For financial cross-sections (one group = one date
   with thousands of names — the primary use case for this library), O(n²) per group per round
   is the dominant training cost and truncation is the standard fix. Pair sampling is the
   alternative if full-depth NDCG matters.
4. Expose `sigma` (sigmoid sharpness) and consider LightGBM's per-query lambda normalization —
   both are cheap knobs that materially affect ranking quality on skewed label distributions.

The same structure (and thus recommendations 1–2) applies to the other pairwise variants
(Pairwise, XeNDCG, YetiRank) and to the joint-path ranking objectives.

### 6.2 GLM (Poisson/Gamma/Tweedie)

Formulas are canonical and the clamping is sensible. Two refinements:

- **Poisson stabilizer**: both LightGBM and XGBoost inflate the Poisson hessian by
  `exp(max_delta_step)` (default e^0.7) to bound the per-leaf Newton step early in training
  when μ is far off; without it, early leaves can swing hard on skewed count data. Cheap,
  well-understood, worth adopting (as `poisson_max_delta_step`, default 0.7).
- **f64 accumulators**: `glm_weighted_target_sum` and the deviance losses accumulate in f32;
  at N ~ 10⁷ with heavy-tailed targets this visibly biases the baseline and the loss telemetry.
  The quantile objective already accumulates loss in f64 — standardize on f64 accumulation
  across all objectives (also fixes `SquaredErrorObjective` if it shares the pattern).

### 6.3 Quantile

The proxy-hessian + per-round empirical-quantile refinement design is right (and refining on
the full dataset under subsampling was the correct call — note it makes quantile *immune* to
bug §1.1's leaf-value component but not its gradient-staleness component). Two notes:

- Split gains under `h_i = w_i` optimize an L2 surrogate of pinball loss; that's standard, but
  after §1.1 is fixed it's worth benchmarking the smoothed-pinball alternative (Huberized
  hessian) which sometimes picks better splits at extreme α.
- `weighted_quantile`'s weighted path sorts the whole leaf (O(n log n)); the unweighted path
  already uses `select_nth_unstable`. A weighted quickselect keeps refinement O(n) per leaf —
  only matters for large leaves × many rounds, measure first.

### 6.4 Multiclass softmax

The K per-class tree builds within a round run serially; class gradient buffers are
precomputed up front (good), so the builds are independent given the shared binned matrix —
`par_iter` over classes is a straightforward K-way speedup for `GBMClassifier` on K ≥ 3
(mind the per-thread histogram arenas — they're already thread-local). Multiclass also still
rejects GLM/quantile objectives (limitations #4) — fine, but the error message should point to
the ranker path that does support them.

## 7. Joint multi-output trainer

- **`u64` categorical bitset caps joint native-categorical at 64 categories** vs the
  single-output path's unbounded `Vec<u8>` bitsets. Validate loudly at fit time (reject or
  fall back per-feature), and document the asymmetry.
- The v0.12.10 numeric split-penalty prefix-sum optimization should be mirrored for joint
  categorical penalties (same issue as §5.3).
- Per-round K-objective gradient computation is serial over outputs — parallelize over K
  (buffers are disjoint), which compounds with §6.1's per-group parallelism for the ranking
  objectives that dominate joint use.
- Verify §1.1 does not apply: the joint round-end uses `walk_tree_into_predictions` over all
  rows, but run the `row_subsample=0.5` divergence A/B on the joint path anyway and pin it
  with a test — this class of bug is invisible to byte-equivalence tests (which compare
  configurations that share the defect).

## 8. DART / GOSS (beyond §1.1–1.2)

- DART's per-round dropout cost is `O(drops × N × tree_depth)` tree walks (subtract + re-add,
  train and validation). With `dart_max_drop=50` defaults on long fits this dwarfs the tree
  build. After §1.2 makes DART usable, profile it; the standard mitigation is capping expected
  drops (`drop_rate × n_trees`) rather than caching per-tree prediction columns (O(N) memory
  per tree is usually worse).
- GOSS's realized-count amplification (rather than symbolic `(1−a)/b`) is a nice correctness
  touch — keep it. Expose `top_rate`/`other_rate` on the estimators if not already surfaced.
